# Карта редиректов для миграции ЖК «Теллерманов сад»

Дата фиксации: 2026-07-07

## 1. Назначение

Документ фиксирует будущую карту переадресаций при переходе от сайта одного ЖК к структуре городского портала «Новостройки Борисоглебска».

Важно: редиректы нельзя включать до полной проверки новых страниц, форм заявок, canonical и sitemap.

## 2. Общая логика

Сейчас старые URL должны оставаться рабочими. Новые URL в `/zhk/tellermanov-sad/` созданы как черновые и закрыты от индексации.

После готовности портала:

1. Старые URL должны вести на новые страницы.
2. В sitemap должны остаться только основные canonical URL.
3. На новых страницах нужно снять `noindex`.
4. Canonical должен указывать на новый URL.
5. Старые страницы должны либо отдавать 301, либо содержать временную HTML-переадресацию, если хостинг не поддерживает серверные редиректы.

## 4. Старые страницы, требующие особого внимания

### `/kvartiry-ot-zastroyschika-borisoglebsk/`

Текущий статус:

- страница закрыта от индексации через `noindex, follow`;
- canonical указывает на `/kvartiry/`;
- URL закрыт в `robots.txt`;
- URL не включен в `sitemap.xml`.

Рекомендация:

1. Не использовать этот URL в новой навигации.
2. При первой возможности удалить ссылку на него из `/karta-sayta/`.
3. После проверки внешних рекламных ссылок заменить страницу на HTML-переадресацию или серверный 301.
4. В новой структуре использовать нейтральный URL `/kvartiry/` или `/zhk/tellermanov-sad/kvartiry/`.

## 5. Варианты для GitHub Pages

GitHub Pages не дает гибко настраивать серверные 301-редиректы без дополнительных решений. Возможные варианты:

### Вариант 1. Оставить старые страницы как рабочие

На старых страницах добавить заметную навигацию на новую структуру, но не делать автоматический редирект.

Плюсы:

- минимальный риск поломки;
- старые рекламные ссылки продолжают работать;
- проще контролировать заявки.

Минусы:

- дублирование контента;
- сложнее SEO-структура.

### Вариант 2. HTML-переадресаторы

На старых страницах оставить минимальный HTML с canonical и ссылкой на новый URL.

Плюсы:

- можно реализовать на GitHub Pages;
- пользователь попадает в новую структуру.

Минусы:

- это не полноценный серверный 301;
- поисковики могут обработать не так идеально, как 301.

### Вариант 3. Переезд на хостинг с 301

Перенести сайт туда, где доступны серверные редиректы: nginx, Apache, Cloudflare Pages, Netlify, Vercel или другой вариант.

Плюсы:

- корректная SEO-миграция;
- гибкая настройка домена и редиректов;
- можно разделить домены и canonical.

Минусы:

- больше технической настройки.

## 6. Рекомендация

Для текущего этапа лучше:

1. Не удалять старые страницы, если на них могут идти рекламные ссылки.
2. Не менять canonical и sitemap без проверки.
3. Держать черновые новые страницы в `noindex`.
4. Доработать новый раздел ЖК.
5. Подготовить новую главную портала.
6. После проверки выбрать доменную стратегию:
   - `tellermanovsad.ru` остается отдельным входом на ЖК;
   - или редиректится на раздел ЖК внутри нового домена портала;
   - или используется как алиас с корректными canonical.
