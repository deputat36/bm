# Безопасность Web3Forms

## Что важно понимать

`access_key` Web3Forms используется на стороне сайта. Для статического сайта это нормальная схема работы сервиса, но ключ не должен давать доступ к CRM, Supabase или другим внутренним системам.

## Что можно хранить в коде

Можно хранить:

- публичный `access_key` Web3Forms;
- URL endpoint Web3Forms;
- текст темы письма;
- названия полей формы.

Нельзя хранить:

- ключи Supabase `service_role`;
- пароли от почты;
- токены Telegram-ботов;
- CRM API key с правами управления клиентами;
- персональные данные тестовых клиентов.

## Как снизить риск спама

1. Не публиковать тестовую страницу в меню сайта.
2. Использовать стандартные антиспам-возможности Web3Forms.
3. При росте спама добавить скрытое поле honeypot.
4. Не писать в рекламе прямую ссылку на `/web3forms-test/`.
5. После запуска отслеживать подозрительные заявки.

## Почему Supabase не подключаем сразу

Прямой public-запрос с сайта в Supabase для записи лидов потребует аккуратной RLS-политики или Edge Function. Для быстрого запуска Web3Forms проще и безопаснее.

Supabase лучше подключать вторым этапом:

- через Edge Function;
- через серверный webhook;
- или через автоматизацию из Web3Forms.
