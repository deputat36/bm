# Модель данных каталога «Новостройки Борисоглебска»

Дата фиксации: 2026-07-07

## Задача

Каталог новостроек должен хранить данные об объектах, застройщиках, формах и SEO в единой структуре. Это нужно, чтобы не дублировать сведения вручную на разных страницах и не расходиться в цифрах по площади, срокам, этажности, документам и условиям покупки.

На первом этапе проект остаётся статическим, поэтому данные хранятся в JSON-файлах внутри папки `data/`.

## Основные сущности

1. Проект портала: общий уровень для всех заявок и страниц. Рекомендуемый идентификатор: `newbuilds-borisoglebsk`.
2. ЖК или объект строительства: отдельный JSON-файл в `data/projects/`.
3. Застройщик: отдельный JSON-файл в `data/builders/`.
4. Форма заявки: HTML-форма с обязательными data-атрибутами для аналитики и CRM.
5. SEO-блок: title, description и h1 для объекта, застройщика или раздела.

## Объект строительства

Файл объекта хранится по пути:

`data/projects/{slug}.json`

Для ЖК «Теллерманов сад»:

`data/projects/tellermanov-sad.json`

Обязательные поля объекта:

- `id` — стабильный внутренний идентификатор;
- `slug` — часть URL;
- `name` — полное название;
- `short_name` — короткое название;
- `city` — город;
- `region` — регион;
- `address` — адрес;
- `builder_id` — связь с застройщиком;
- `builder_name` — название застройщика для вывода;
- `status` — статус объекта;
- `sales_status` — статус продаж и важная оговорка;
- `is_active` — показывать ли объект в каталоге;
- `is_featured` — выделять ли объект на главной или в каталоге.

## Характеристики объекта

Рекомендуемые поля:

- `class` — класс жилья, если подтверждён;
- `wall_material` — материал стен;
- `floors` — этажность;
- `entrances` — количество подъездов;
- `apartments_total` — общее количество квартир;
- `ceiling_height` — высота потолков;
- `handover` — планируемый срок сдачи;
- `keys_until` — срок передачи ключей, если указан;
- `area_min` и `area_max` — общий диапазон площадей.

## Квартирография

Квартирография хранится в массиве `apartment_types`.

Для каждого типа квартиры нужны поля:

- `type` — машинный тип: studio, one-room, two-room, three-room;
- `title` — человекочитаемое название;
- `rooms` — количество комнат;
- `count` — количество квартир такого типа;
- `area_min` — минимальная площадь;
- `area_max` — максимальная площадь.

## Документы и источники

Рекомендуемые поля:

- `project_declaration` — номер проектной декларации;
- `nash_dom_rf_id` — ID объекта на наш.дом.рф;
- `building_permit` — номер разрешения на строительство;
- `building_permit_date` — дата разрешения;
- `project_declaration_date` — дата проектной декларации;
- `source_status` — статус источника данных;
- `source_note` — пояснение по источнику.

## Застройщики

Файлы застройщиков хранятся по пути:

`data/builders/{slug}.json`

Рекомендуемые поля:

- `id` — внутренний идентификатор;
- `slug` — часть URL;
- `name` — юридическое название;
- `brand_name` — коммерческий бренд;
- `city` — город;
- `region` — регион;
- `logo` — путь к логотипу;
- `description` — короткое описание;
- `projects` — массив связанных объектов;
- `is_active` — показывать ли застройщика на сайте;
- `seo` — SEO-блок страницы.

## Формы заявок

Каждая форма должна передавать стабильные идентификаторы:

- `data-lead-form` — признак формы заявки;
- `data-lead-type` — тип заявки;
- `data-project` — текстовое название проекта;
- `data-project-id` — идентификатор проекта;
- `data-project-name` — название проекта для CRM;
- `data-complex` — название ЖК, если заявка по конкретному объекту;
- `data-complex-id` — идентификатор ЖК;
- `data-form-id` — идентификатор формы;
- `data-success-message` — сообщение после отправки.

Для общей формы каталога использовать тип `portal_selection`.

Для формы конкретного ЖК использовать тип `project_consultation`.

## Статусы данных

Чтобы не смешивать публичные и рабочие сведения, каждому источнику желательно назначать статус:

- `public` — данные из открытых публичных источников;
- `official` — официально предоставленные материалы;
- `internal` — внутренняя рабочая информация, не для публикации;
- `preliminary` — предварительные данные, требующие проверки;
- `unknown` — данных пока нет.

Для публичных страниц использовать только проверенные данные или явно писать, что информация предварительная и требует уточнения.

## Правила публикации

1. Не публиковать закрытую рабочую информацию без отдельного решения.
2. Не использовать формулировки «официальный сайт», «официальный представитель», «цены от застройщика», если это не подтверждено юридически.
3. Не фиксировать цену как гарантированную.
4. Для неподтверждённых условий использовать формулировки: «может быть доступно», «необходимо уточнить», «информация справочная».
5. Для каждой страницы сохранять понятный источник данных.
6. При изменении цен, сроков или статуса продаж обновлять JSON и страницы одновременно.

## Следующий технический шаг

После утверждения модели данных можно добавить:

1. `data/projects/index.json` — общий список активных объектов;
2. `data/builders/index.json` — общий список застройщиков;
3. загрузку карточек в `/novostroyki/` из JSON;
4. генерацию таблицы сравнения из тех же данных;
5. единые компоненты карточек для каталога, главной и раздела сравнения.
