# Приватность локального fallback заявки

Дата обновления: 2026-07-15

## Назначение

Портал использует Web3Forms как текущий внешний канал доставки. Локальный fallback нужен только для технического режима, когда в конфигурации отсутствуют все внешние направления.

Он не должен превращать браузер пользователя в хранилище заявок и персональных данных.

## Исправленный риск

Ранее при отсутствии внешнего направления в `localStorage` под ключом:

```text
newbuildsBorisoglebskLeadsDraft
```

мог сохраняться полный объект формы. Он включал имя, телефон, комментарий, параметры покупки, согласия и технические данные.

У записи не было ограниченного срока хранения и лимита количества элементов.

## Новое поведение

При загрузке страницы общий runtime удаляет старый ключ:

```text
newbuildsBorisoglebskLeadsDraft
```

Если отсутствуют и `LEAD_ENDPOINT`, и активная отправка Web3Forms, сохраняется только обезличенная техническая квитанция под ключом:

```text
newbuildsBorisoglebskOfflineReceipts
```

Максимум хранится пять последних квитанций.

## Поля квитанции

Разрешены только:

```text
lead_type
form_id
project_id
residential_complex_id
qualification_status
lead_source
placement
created_at
contact_data_stored=false
```

Не сохраняются:

```text
имя
телефон
email
бюджет
комментарий или вопрос
интерес и комнатность
способ покупки
срок покупки
текст согласия
user agent
client_fixation_id
полный tracking
полный payload формы
```

## Внешняя доставка

Если настроен хотя бы один внешний канал, privacy-override вызывает исходную функцию `sendLead` без изменения:

```text
Web3Forms настроен → исходная отправка
LEAD_ENDPOINT настроен → исходная отправка
honeypot заполнен → исходная блокировка
внешних каналов нет → только обезличенная квитанция
```

Web3Forms, адрес получателя, payload письма и страница благодарности этим изменением не затрагиваются.

## Сообщение пользователю

Локальная квитанция не означает получение заявки специалистом. В тестовом fallback пользователь должен видеть рекомендацию связаться по телефону.

Поле:

```text
contact_data_stored=false
```

явно запрещает интерпретировать квитанцию как сохранённый контакт.

## Dry-run

Ручной dry-run остаётся отдельным явно включаемым тестовым режимом:

```text
lead_test=dry-run
test_ack=1
```

Он не является рабочим fallback и должен использовать только тестовые данные.

## Автоматическая проверка

Запуск:

```bash
node tools/validate-offline-draft-privacy.mjs
```

Guard проверяет:

- удаление legacy-ключа;
- максимум пять квитанций;
- разрешённый список полей;
- отсутствие персональных и избыточных данных;
- сохранение исходного внешнего delivery-пути;
- отдельную обработку honeypot;
- запуск privacy-override до dry-run;
- отсутствие старой операции `saved.push(data)`.

## Что не реализовано

Изменение не добавляет:

- серверный offline queue;
- повторную автоматическую отправку;
- CRM;
- Supabase;
- хранение реальных обращений в GitHub;
- обещание, что локальная квитанция будет обработана специалистом.
