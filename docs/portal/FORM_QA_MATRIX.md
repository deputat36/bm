# Матрица ручной проверки форм

Дата обновления: 2026-07-14

## Назначение

Матрица формирует единый набор безопасных сценариев для проверки всех активных форм портала на:

- компьютере;
- Android;
- iPhone.

Проверка выполняется только в сочетании безопасного dry-run и локального analytics debug. Тестовые данные не отправляются в Web3Forms, email, CRM, Google Analytics или Яндекс Метрику.

Источник сценариев:

```text
data/qa/form-scenarios.json
```

## Состав матрицы

Зарегистрировано 14 сценариев:

```text
7 primary — короткие формы
7 detailed — подробные формы
```

Страницы:

```text
/
/catalog/
/catalog/prostornaya-4a/
/catalog/aerodromnaya-18g/
/catalog/sennaya-76/
/contacts/
/ipoteka/
```

Для каждой страницы проверяются обе формы.

## Безопасные параметры

Каждая готовая ссылка содержит одновременно:

```text
lead_test=dry-run
analytics_test=debug
test_ack=1
```

Без всех трёх параметров локальный analytics debug не включается.

Якоря:

```text
primary  → #quick-lead
detailed → #lead
```

## Проверка матрицы

Команда:

```bash
npm run validate:form-qa
```

Validator проверяет:

- ровно 14 сценариев;
- ровно 7 primary и 7 detailed;
- уникальность ID сценариев и `form_id`;
- существование страницы и формы;
- совпадение `lead_type`;
- правильный якорь;
- размещение primary внутри `data-primary-lead`;
- отсутствие detailed внутри primary-контейнера;
- совпадение объектного `data-complex-id`;
- поддержку событий из реестра аналитики;
- запрет реальной отправки и публичных тестовых ссылок.

## Генератор готовых ссылок

Markdown:

```bash
npm run qa:form-links
```

JSON:

```bash
npm run qa:form-links -- --format=json
```

CSV:

```bash
npm run qa:form-links -- --format=csv
```

Только короткие формы:

```bash
npm run qa:form-links -- --role=primary
```

Только подробные формы:

```bash
npm run qa:form-links -- --role=detailed
```

Параметры можно совмещать:

```bash
npm run qa:form-links -- --format=csv --role=primary
```

## Содержимое выгрузки

Для каждого сценария выводятся:

```text
id
label
page_path
form_id
form_role
lead_type
object_id
anchor
url
devices
required_events
optional_events
```

Готовые ссылки предназначены только для технического QA. Их нельзя размещать в публичных постах, рекламе, меню или контенте сайта.

## Обязательная последовательность событий

После открытия формы и отправки dry-run ожидаются:

```text
lead_form_view
→ lead_form_start
→ lead_submit
→ lead_submit_classified
→ lead_thankyou_view
```

Дополнительно могут проверяться:

```text
lead_cta_click
lead_postsubmit_action
```

Локально моделируемые submit-события имеют:

```text
simulated=true
offline=true
blocked=false
```

## Чек-лист одного сценария

1. Открыть готовую ссылку из матрицы.
2. Убедиться, что появились красная панель dry-run и синяя панель analytics debug.
3. Проверить, что страница прокручена к ожидаемой форме.
4. Проверить видимость и удобство формы.
5. Начать заполнение и убедиться в появлении `lead_form_start`.
6. Использовать только тестовое имя и тестовый телефон.
7. Заполнить обязательные поля.
8. Отправить форму.
9. Убедиться, что данные не отправлены наружу.
10. Проверить `lead_submit` и `lead_submit_classified`.
11. Проверить правильный `form_role`.
12. Проверить переход на тестовую страницу благодарности.
13. Проверить `lead_thankyou_view`.
14. Выполнить одно действие после отправки и проверить `lead_postsubmit_action`.
15. Скопировать JSON локального журнала.
16. Зафиксировать устройство, браузер, результат и найденные проблемы.
17. Очистить журнал перед следующим независимым сценарием.

## Что проверять на смартфоне

### Общие требования

- форма полностью помещается по ширине;
- текст не выходит за экран;
- кнопка доступна без горизонтальной прокрутки;
- поля не перекрываются мобильной панелью;
- фокус видим;
- ошибки читаемы;
- после отправки открывается корректная тестовая страница.

### Android

Проверить минимум в актуальном Chrome:

- телефонную клавиатуру для поля телефона;
- выбор вариантов в `select`;
- прокрутку при открытой клавиатуре;
- копирование JSON из debug-панели.

### iPhone

Проверить минимум в актуальном Safari:

- отсутствие автоматического масштабирования полей;
- доступность кнопки при открытой клавиатуре;
- прокрутку к ошибочному полю;
- работу системного буфера обмена;
- сохранение режима при переходе на `/spasibo/`.

## Фиксация результата

Рекомендуемый формат одной записи:

```text
Дата:
Устройство:
ОС:
Браузер:
Сценарий:
form_id:
form_role:
Результат: passed | failed | blocked
События:
Проблемы:
Ссылка на скриншот или JSON:
Комментарий:
```

Статус `passed` можно ставить только после прохождения всей последовательности до страницы благодарности.

## Portal guards

CI выполняет:

```text
npm run validate:form-qa
npm run qa:form-links -- --format=json
```

Контрольная генерация требует:

```text
14 сценариев
7 primary
7 detailed
```

Изменения в `data/qa/**`, генераторе, validator или формах запускают Portal guards.

## Ограничения

Матрица и автоматические guards не заменяют реальное тестирование на устройствах.

До ручной проверки нельзя утверждать, что:

- все формы проверены на Android;
- все формы проверены на iPhone;
- все мобильные браузеры отображают формы одинаково;
- внешняя система аналитики получает события;
- реальная заявка доставляется специалисту.

Реальная отправка формы остаётся отдельным действием и требует согласования.
