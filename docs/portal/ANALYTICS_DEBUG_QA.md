# Локальная проверка аналитики

Дата обновления: 2026-07-14

## Назначение

Режим позволяет проверить полную конверсионную воронку на компьютере, Android или iPhone без:

- отправки тестовой заявки в Web3Forms, email или CRM;
- записи событий в Google Analytics;
- записи целей в Яндекс Метрику;
- смешивания тестовых действий с реальным трафиком.

События сохраняются только в `sessionStorage` текущей вкладки.

## Включение

Нужны одновременно три параметра:

```text
lead_test=dry-run
analytics_test=debug
test_ack=1
```

Пример для главной:

```text
/?lead_test=dry-run&analytics_test=debug&test_ack=1#quick-lead
```

Пример для каталога:

```text
/catalog/?lead_test=dry-run&analytics_test=debug&test_ack=1#quick-lead
```

Пример для объекта:

```text
/catalog/prostornaya-4a/?lead_test=dry-run&analytics_test=debug&test_ack=1#quick-lead
```

Локальный analytics debug не включается без `lead_test=dry-run`. Это защищает от случайной отправки формы во время технической проверки.

## Интерфейс

После включения появляются:

1. Красная панель dry-run — подтверждает, что заявка не отправляется наружу.
2. Синяя панель локальной аналитики — показывает число и последние события.

В панели доступны действия:

- `Копировать JSON` — копирует журнал текущей вкладки;
- `Очистить` — удаляет локальный журнал;
- `Скрыть` — сворачивает панель, не останавливая запись.

Ключ хранения:

```text
newbuildsBorisoglebskAnalyticsDebugEvents
```

Лимит:

```text
100 последних событий
```

## Ожидаемая последовательность

При проверке короткой формы:

```text
lead_form_view
→ lead_form_start
→ lead_submit
→ lead_submit_classified
→ lead_thankyou_view
→ lead_postsubmit_action
```

`lead_submit` и `lead_submit_classified` в журнале имеют:

```text
simulated=true
offline=true
blocked=false
```

Они являются локальной моделью событий и не попадают в рабочие счётчики.

## Проверка CTA

Перед формой можно нажать размеченную кнопку. В журнале должно появиться:

```text
lead_cta_click
```

Проверяются:

```text
action
placement
object_id
link_url
```

## Проверка роли формы

Для короткой формы:

```text
form_role=primary
```

Для подробной формы:

```text
form_role=detailed
```

На каждой из семи основных страниц работает одна пара primary/detailed.

## Проверка страницы благодарности

Параметры режима автоматически сохраняются при переходе после dry-run.

На `/spasibo/` локально моделируются:

```text
lead_thankyou_view
lead_postsubmit_action
```

Можно проверить:

- возврат к выбранному объекту;
- открытие каталога;
- повторную заявку;
- нажатие на телефон.

## Защита данных

Перед сохранением локального события удаляются:

```text
name
phone
phone_normalized
budget
comment
question
consent_text
client_fixation_id
user_agent
access_key
fields_json
message
```

Журнал нельзя использовать как источник персональных данных или как подтверждение реальной доставки заявки.

## Ручной чек-лист

Для каждой проверяемой страницы:

1. Открыть URL с тремя тестовыми параметрами.
2. Убедиться, что видны красная и синяя панели.
3. Проверить `lead_form_view`.
4. Начать заполнение и проверить `lead_form_start`.
5. Заполнить обязательные поля тестовыми данными.
6. Отправить форму.
7. Проверить локальные `lead_submit` и `lead_submit_classified`.
8. Убедиться в переходе на тестовую страницу благодарности.
9. Проверить `lead_thankyou_view`.
10. Нажать одно post-submit действие.
11. Скопировать JSON и сохранить результат QA.
12. Очистить журнал перед следующим отдельным сценарием.

## Выключение

Кнопка `Выключить тест` удаляет:

```text
lead_test
analytics_test
test_ack
```

После выключения страницу нужно использовать как обычную. Реальную отправку заявки следует выполнять только после отдельного согласования.

## Автоматическая проверка

Команда:

```bash
npm run validate:analytics-debug
```

Guard проверяет:

- тройное условие включения;
- загрузку debug-скрипта раньше конверсионного скрипта;
- локальное хранение и лимит журнала;
- удаление персональных и ограниченных полей;
- возврат до `dataLayer`, `gtag` и Метрики;
- сохранение параметров при переходе на `/spasibo/`;
- локальную симуляцию submit, thank-you и post-submit;
- отсутствие публичных ссылок, включающих тестовый режим.

## Ограничения

Режим не подтверждает:

- наличие и корректность установленного рабочего счётчика;
- поступление событий в интерфейс Google Analytics или Яндекс Метрики;
- реальную доставку заявки;
- работу внешнего CRM endpoint;
- отсутствие особенностей конкретного браузера.

После локального QA потребуется отдельная проверка рабочего счётчика в debug-интерфейсе выбранной системы аналитики.
