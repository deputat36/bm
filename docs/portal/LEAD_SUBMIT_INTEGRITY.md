# Целостность отправки заявок

Дата обновления: 2026-07-15

## Назначение

Этот контур защищает пользователя и менеджера от ложного результата формы и случайной повторной отправки.

Исправлены два runtime-сценария:

1. внешняя доставка уже завершилась успешно, но запись в `localStorage` завершилась ошибкой;
2. второй `submit` был вызван до завершения первого сетевого запроса.

Без отдельной защиты первый сценарий мог показать пользователю сообщение «Не удалось отправить заявку», хотя письмо уже было отправлено. Это повышало риск повторной заявки.

Отключение кнопки не является достаточной защитой второго сценария: событие `submit` можно вызвать клавишей Enter, скриптом или другим элементом формы.

## Safe storage

Прямой доступ к `localStorage` разрешён только внутри двух обёрток:

```text
safeStorageGet
safeStorageSet
```

Если браузер запрещает storage, переполнена квота или доступ блокируется настройками приватности:

- сбор tracking-контекста продолжает работать в памяти текущей страницы;
- успешная внешняя отправка не превращается в ошибку интерфейса;
- отсутствие записи `lastLead` не отменяет факт доставки;
- исключение storage не прерывает инициализацию формы.

## Офлайн-режим

Статус `offline=true` разрешён только после успешной записи локального черновика.

Алгоритм:

```text
прочитать массив черновиков через safeStorageGet
→ добавить заявку
→ сохранить через safeStorageSet
→ только затем вернуть offline=true
```

Если локальный черновик сохранить невозможно, форма показывает ошибку отправки. Она не утверждает, что заявка сохранена.

## In-flight lock

После прохождения HTML-валидации форма получает:

```text
data-submitting="true"
aria-busy="true"
```

Пока первый запрос не завершён, следующий `submit` немедленно прекращается.

Lock снимается в `finally`, поэтому повторная попытка снова доступна после сетевой ошибки.

Кнопка отправки остаётся дополнительной визуальной защитой, но не является единственным механизмом блокировки.

## Honeypot

Honeypot-заявка:

- не отправляется в Web3Forms;
- получает `blocked=true` в техническом событии;
- не переходит на страницу благодарности;
- не записывается как `lastLead`;
- не может подменить контекст последней настоящей заявки.

Blocked-события по-прежнему доступны для технического контроля, но исключаются из канонической метрики заявок фильтром `blocked=false`.

## Порядок успешной отправки

```text
HTML validation
→ установить in-flight lock
→ собрать данные
→ выполнить delivery
→ отправить техническое событие
→ очистить форму
→ проверить blocked
→ сохранить безопасный lastLead
→ перейти на страницу благодарности или показать результат
→ снять lock
```

## Автоматическая проверка

Файл:

```text
tools/validate-lead-submit-integrity.mjs
```

Workflow:

```text
Lead submit integrity guard
```

Guard проверяет:

- прямой `localStorage.getItem` существует только внутри `safeStorageGet`;
- прямой `localStorage.setItem` существует только внутри `safeStorageSet`;
- tracking использует безопасные обёртки;
- офлайн-статус возвращается только после подтверждённой записи;
- повторный submit блокируется до сетевого запроса;
- `aria-busy` устанавливается и снимается;
- lock снимается в `finally`;
- `trackLeadEvent` вызывается и для blocked-попытки;
- `saveLastLead` выполняется только после проверки `result.blocked`.

## Ограничения проверки

CI проверяет структуру и синтаксис кода, но не подтверждает:

- фактическую доставку письма;
- поведение конкретного браузера с запрещённым storage;
- физический двойной клик на Android или iPhone;
- работу Web3Forms в производственном домене;
- отсутствие дублей на серверной стороне.

Серверная идемпотентность и объединение повторных обращений остаются задачей будущего endpoint или CRM.
