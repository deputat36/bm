# Аудит третьей очереди legacy-страниц Просторной 4А

Дата: 2026-07-13

Репозиторий: `deputat36/bm`

## Цель

Завершить разбор старых объектных URL, убрать старый домен и брендинг, исключить неподтверждённые данные из поисковой выдачи и определить судьбу полезного содержания.

Основная карточка объекта:

```text
/catalog/prostornaya-4a/
```

|---|---|---|---|

| `/infrastruktura/` | transition_page | retire | `/catalog/prostornaya-4a/` |

## Что изменено на страницах

```text
noindex,follow
удалён canonical старого домена
удалён брендинг Этажи и BM Group
удалена старая объектная навигация
удалены устаревшие CTA и формы
удалена FAQ-разметка с неподтверждёнными ответами
добавлена явная ссылка на актуальный раздел
нет meta refresh и JavaScript redirect
```

## Решения по содержанию

### retain_content

`/galereya/`

Рендеры и изображения не удаляются из репозитория автоматически. Перед переносом в карточку объекта нужно проверить:

1. источник каждого файла;
2. право публикации;
3. соответствие актуальной версии проекта;
4. отсутствие вводящих в заблуждение деталей;
5. подпись о том, что визуализация может отличаться от результата строительства.

`/proektnaya-deklaratsiya-36-001139/`

Полезны номер документа, даты и реквизиты, но их необходимо повторно сверить с первичным источником. После проверки данные должны храниться в профиле проекта и выводиться в одном документном блоке карточки.

`/faq/`

Вопросы нужно разделить:

- общие вопросы о покупке новостройки — в `/guides/`;
- вопросы по конкретному дому — в карточку Просторной 4А;
- ответы о ценах, наличии, сроках и ипотеке — только после актуальной проверки.

### retire

`/infrastruktura/`

Старая страница была шаблонной и не содержала проверенных расстояний, маршрутов или самостоятельного материала. Отдельный URL не нужен.

`/hod-stroitelstva/`

Старая страница не содержала датированных фотоотчётов. Будущие обновления должны публиковаться в карточке объекта или новостях портала с датой и источником.

### redirect

`/zastroyschik/`

Старый URL должен вести в нейтральный раздел `/developers/`. Отдельную карточку юридического лица можно создать только после проверки названия, реквизитов, связи с объектом и источников.

## Машиночитаемый контроль

Все маршруты внесены в:

```text
data/migration/legacy-routes.json
```

Для каждого маршрута теперь обязательно поле:

```text
migration_action: redirect | retain_content | retire
```

Для `retain_content` обязательно:

```text
content_migration_status: pending | migrated | not_required
```

Команда проверки:

```bash
npm run validate:legacy
```

Валидатор не позволит пометить маршрут готовым к редиректу, пока полезное содержание со статусом `pending` не перенесено.

## Текущий результат

Все известные корневые legacy-страницы объекта переведены в безопасный переходный режим. Серверные редиректы не включены. Следующий шаг — перенос проверенного содержания из категорий `retain_content`, затем проверка основной карточки и платформы размещения.
