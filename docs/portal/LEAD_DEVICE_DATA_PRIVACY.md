# Приватность device data в заявке

Статус: техническая защита включена, ручная проверка реального письма не выполнена.

## Риск

Базовый сборщик формы добавлял `navigator.userAgent` в полный объект заявки. Поле не использовалось менеджером, но могло попадать в `fields_json` Web3Forms или в будущий endpoint.

User agent может содержать сведения о браузере, операционной системе и устройстве. Для обработки консультационной заявки эти данные не нужны.

## Реализованная защита

`assets/js/form-accessibility.js` устанавливает privacy-wrapper на всех активных страницах форм.

Очистка выполняется в двух точках:

1. сразу после `collectFormData`;
2. непосредственно перед `sendLead`.

Из объекта удаляется:

```text
user_agent
```

Остальные поля заявки, контакты клиента, объект, форма, согласие и разрешённая атрибуция не меняются.

## Каналы

После установки wrapper `user_agent` не должен попадать в:

- Web3Forms;
- `fields_json`;
- читаемый текст письма;
- будущий `LEAD_ENDPOINT`;
- аналитику;
- `lastLead`;
- локальную офлайн-квитанцию;
- dry-run evidence.

## CI

Проверка `tools/validate-lead-device-data-privacy.mjs` подтверждает:

- наличие wrapper после сбора формы;
- повторную очистку перед delivery;
- загрузку runtime на активных страницах;
- отсутствие top-level поля в Web3Forms;
- запрет `user_agent` в аналитическом реестре;
- установку privacy раньше enhancement форм.

## Ограничения

Автоматический guard не заменяет ручную проверку фактического письма и Network payload. Реальные персональные данные для теста использовать нельзя.
