# Доступность и мобильная проверка форм

Дата: 2026-07-13

## Цель

Все рабочие формы портала должны быть удобны не только визуально, но и при использовании клавиатуры, экранного диктора и мобильного устройства.

## Автоматическое улучшение

Файл:

```text
assets/js/form-accessibility.js
```

Он подключается через `assets/js/schema.js` на каждой странице, где присутствует `form[data-lead-form]`.

Для формы автоматически настраиваются:

```text
aria-busy
aria-invalid
aria-controls
role=status
aria-live=polite
aria-atomic=true
inputmode=tel
enterkeyhint
```

Состояние `aria-busy` синхронизируется с блокировкой кнопки отправки. При невалидном поле фокус переводится на первое поле с ошибкой. После исправления `aria-invalid` снимается.

## Автоматическая проверка

Команда:

```bash
npm run validate:a11y
```

Проверка охватывает все страницы со статусом `ready` или `published` из `data/pages/index.json`, на которых есть лид-формы.

Контролируются:

```text
подключение main.js
подключение schema.js
наличие поля имени
наличие поля телефона
наличие области статуса
наличие кнопки отправки
загрузка form-accessibility.js
наличие основных ARIA-механизмов
```

Проверка включена в `.github/workflows/portal-guards.yml`.

## Ручной мобильный сценарий

Для каждой рабочей формы нужно проверить экран шириной 320, 360, 390 и 430 пикселей.

1. Открыть страницу без сохранённых данных и рекламных меток.
2. Убедиться, что форма не выходит за границы экрана.
3. Проверить появление мобильной панели действий.
4. Нажать кнопку перехода к форме и убедиться, что первое поле получает фокус.
5. Попытаться отправить пустую форму.
6. Проверить фокус на первом обязательном поле и отсутствие скачка страницы.
7. Заполнить имя и телефон экранной клавиатурой.
8. Проверить, что для телефона открывается цифровая или телефонная клавиатура.
9. Проверить выбор объекта, способа покупки и срока.
10. Проверить обязательное согласие на обработку данных.
11. Во время отправки убедиться, что кнопка заблокирована и повторный клик невозможен.
12. Проверить страницу благодарности либо сообщение об ошибке.
13. Вернуться назад и убедиться, что форма снова доступна.

## Проверка с клавиатуры

Порядок `Tab` должен быть логичным:

```text
поле имени → телефон → параметры заявки → комментарий → согласие → кнопка
```

Фокус должен быть видимым. Пользователь должен иметь возможность открыть ссылки на политику и согласие, вернуться к форме и продолжить заполнение.

## Проверка экранным диктором

Нужно подтвердить, что:

```text
подпись каждого поля читается до значения
обязательные поля определяются как обязательные
ошибка поля объявляется после неудачной отправки
статус отправки объявляется без ручного поиска
заблокированная кнопка определяется как недоступная
```

## Ограничение

Автоматические улучшения не заменяют ручную проверку в реальном браузере и на смартфоне. Контролируемая отправка заявки проводится только с явно тестовыми именем, телефоном и комментарием, поскольку текущая форма отправляет сообщение на рабочий канал.
