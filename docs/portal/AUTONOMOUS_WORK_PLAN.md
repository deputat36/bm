# План самостоятельной работы над порталом

Дата фиксации: 2026-07-09

Репозиторий: `deputat36/bm`.

## 1. Режим работы

Дальше работу по порталу ведём в этом диалоге как в управляющем и рабочем центре.

Цель:

```text
самостоятельно маленькими безопасными шагами улучшать сайт novostroyki-borisoglebsk.ru и превращать его в полноценный портал новостроек Борисоглебска
```

Каждый шаг должен быть:

1. небольшим;
2. проверяемым;
3. обратимым;
4. юридически безопасным;
5. не ломающим текущий работающий сайт.

## 2. Что можно делать самостоятельно без дополнительного согласования

Можно выполнять сразу:

1. Создавать и обновлять документацию в `docs/portal/`.
2. Создавать черновые страницы с `noindex,follow`.
3. Дорабатывать `/catalog/` и карточки объектов, если они остаются закрытыми от индексации.
4. Добавлять черновые JSON-данные с `is_public_ready: false`.
5. Добавлять пометки `Данные уточняются`.
6. Улучшать формы заявок без изменения адресатов и ключей отправки.
7. Улучшать внутреннюю структуру черновых страниц.
8. Создавать безопасные тексты без обещаний цены, брони и официального статуса.
9. Подготавливать карту миграции URL.
10. Подготавливать будущий sitemap-preview, не заменяя рабочий `sitemap.xml`.
11. Создавать справочные страницы в черновом режиме.
12. Добавлять страницы застройщиков в черновом режиме.

## 3. Что нельзя делать без отдельного подтверждения владельца

Не выполнять без прямого подтверждения:

```text
замена корневого index.html
изменение sitemap.xml
изменение robots.txt
изменение canonical на старых страницах
удаление старых страниц
массовые редиректы
открытие новых страниц для индексации
публикация цен
публикация планировок как официальных без источника
публикация неподтверждённых данных по объектам
изменение ключей отправки форм
изменение телефона на сайте
изменение юридически чувствительных формулировок в сторону усиления рекламных обещаний
```

## 4. Рабочая стратегия

### Фаза 1. Навести порядок в управлении проектом

Статус: в работе.

Задачи:

1. Поддерживать `CONTROL_CENTER.md`.
2. Поддерживать `STATUS.md`.
3. Поддерживать `ROADMAP.md`.
4. Поддерживать `AUDIT_2026-07-09.md`.
5. Поддерживать этот файл.

Цель фазы:

```text
чтобы все следующие изменения были связаны с общей стратегией портала
```

### Фаза 2. Данные объектов

Задачи:

1. Создать `docs/portal/OBJECTS_DATABASE.md`.
2. Зафиксировать таблицу объектов первой очереди:
   - Просторная 4А;
   - Аэродромная 18Г;
   - Сенная 76;
   - проекты ПСК;
   - проекты BM Group;
   - другие новостройки с 2018 года.
3. Создать черновые JSON-файлы:
   - `data/projects/prostornaya-4a.json`;
   - `data/projects/aerodromnaya-18g.json`;
   - `data/projects/sennaya-76.json`.
4. Расширить `data/projects/index.json` черновыми объектами.

Правило:

```text
если данные не подтверждены, использовать verification_status: draft и is_public_ready: false
```

### Фаза 3. Каталог

Задачи:

1. Доработать `/catalog/index.html`.
2. Добавить фильтры или визуальные группы:
   - строится;
   - построен;
   - данные уточняются;
   - по адресу;
   - по застройщику.
3. Усилить карточки объектов.
4. Добавить дисклеймеры и статус проверки данных.
5. Сделать каталог удобным для будущей главной.

### Фаза 4. Карточки объектов

Задачи:

1. Доработать `/catalog/prostornaya-4a/`.
2. Доработать `/catalog/aerodromnaya-18g/` как безопасную черновую страницу.
3. Доработать `/catalog/sennaya-76/` как безопасную черновую страницу.
4. Сделать единый шаблон карточки объекта:
   - адрес;
   - статус;
   - застройщик;
   - сроки;
   - документы;
   - квартиры;
   - ипотека;
   - плюсы;
   - что требует проверки;
   - форма заявки.

### Фаза 5. Застройщики

Задачи:

1. Создать или доработать `/developers/`.
2. Создать черновые страницы:
   - `/developers/bm-group/`;
   - `/developers/psk/`.
3. Добавить связи застройщик → объекты.
4. Не утверждать официальный статус без источников.

### Фаза 6. Лидогенерация

Задачи:

1. Создать `docs/portal/LEADS_AND_CRM.md`.
2. Описать все типы форм.
3. Проверить все `data-lead-type`.
4. Проверить, что формы передают объект и источник.
5. Добавить сценарии обработки заявок.
6. Подготовить тексты кнопок под разные сценарии.

### Фаза 7. Юридическая безопасность

Задачи:

1. Создать `docs/portal/LEGAL_RULES.md`.
2. Зафиксировать стоп-лист.
3. Зафиксировать разрешённые формулировки.
4. Добавить шаблон дисклеймера.
5. Проверить тексты новых страниц.

### Фаза 8. Главная страница

Выполнять только после готовности каталога.

Порядок:

1. Довести `/portal-preview/` до полноценной будущей главной.
2. Проверить юридические формулировки.
3. Проверить формы.
4. Проверить ссылки.
5. Подготовить копию будущего `index.html` как отдельный файл или черновик.
6. После подтверждения владельца заменить корень.

## 5. Формат каждого самостоятельного цикла

Каждый цикл работы должен проходить так:

```text
1. Проверить текущие файлы.
2. Выбрать одну небольшую задачу.
3. Внести изменения.
4. Сделать один маленький коммит.
5. Проверить затронутые файлы.
6. Дать короткий отчёт.
7. Зафиксировать следующий шаг.
```

## 6. Приоритет ближайших самостоятельных задач

1. Создать `OBJECTS_DATABASE.md`.
2. Создать `URL_MIGRATION_MAP.md`.
3. Создать `LEADS_AND_CRM.md`.
4. Создать `LEGAL_RULES.md`.
5. Расширить `data/projects/index.json` безопасными черновыми объектами.
6. Доработать `/catalog/index.html`.
7. Доработать карточки `/catalog/aerodromnaya-18g/` и `/catalog/sennaya-76/`.
8. Подготовить `/developers/`.
9. Доработать `/portal-preview/` как будущую главную.

## 7. Критерии готовности к замене главной

Главную можно заменить только когда:

1. `/catalog/` готов как основной раздел.
2. Есть минимум 3 карточки объектов.
3. Все новые страницы юридически безопасны.
4. Формы заявок проверены.
5. Будущий sitemap подготовлен.
6. Карта миграции URL готова.
7. Владелец подтвердил замену корневого `index.html`.

## 8. Короткое правило

```text
Сначала строим безопасный каталог в noindex.
Потом наполняем объектами.
Потом проверяем юридически.
Потом только переносим главную и открываем SEO.
```
