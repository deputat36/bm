## 2. Главный вывод

Проект уже начал переход от сайта одного объекта к городскому порталу, но публичная структура ещё не завершила миграцию.

Сейчас есть две параллельные логики:

1. Старый публичный контур — сайт ЖК «Теллерманов сад» / Просторная 4А.
2. Новый портальный контур — каталог всех новостроек Борисоглебска.

Целевая стратегия правильная:

```text
novostroyki-borisoglebsk.ru = городской портал-каталог всех новостроек Борисоглебска
```

Но корневой `index.html` ещё не соответствует этой стратегии.

## 4. Критические проблемы

### 4.1. Корневая страница всё ещё про один ЖК

`index.html` содержит:

- title про ЖК «Теллерманов сад»;
- description про дом на Просторной 4А;
- canonical на `https://tellermanovsad.ru/`;
- OG URL на `https://tellermanovsad.ru/`;
- микроразметку `Residence` одного объекта;
- брендовый блок с логотипом BM Group;
- меню старого сайта одного ЖК;
- hero одного объекта.

Это конфликтует с текущей стратегией портала.

### 4.2. Sitemap всё ещё относится к старому домену

`sitemap.xml` содержит URL домена:

```text
https://tellermanovsad.ru/
```

Для нового портала это нельзя считать финальным состоянием.

### 4.3. Robots указывает sitemap старого домена

`robots.txt` содержит:

```text
Sitemap: https://tellermanovsad.ru/sitemap.xml
```

После миграции нужно будет заменить на:

```text
Sitemap: https://novostroyki-borisoglebsk.ru/sitemap.xml
```

Но делать это нужно только после подготовки новой структуры.

### 4.4. Данные каталога пока неполные

`data/projects/index.json` содержит только один объект:

```text
ЖК Теллерманов сад / Просторная 4А
```

Для портала нужны минимум:

```text
Просторная 4А
Аэродромная 18Г
Сенная 76
проекты ПСК
проекты BM Group
остальные новостройки с 2018 года
```

### 4.5. Есть дублирующая архитектура URL

В проекте одновременно существуют или описаны две схемы:

Старая / промежуточная:

```text
/novostroyki/
/zhk/
/zastroyschiki/
/spravochnik/
/novosti/
/sravnenie/
```

Новая целевая:

```text
/catalog/
/catalog/prostornaya-4a/
/catalog/aerodromnaya-18g/
/catalog/sennaya-76/
/developers/
/guides/
/news/
```

Нужно постепенно приводить сайт к новой схеме и не развивать параллельно две разные структуры.

## 5. Юридические риски

1. Сайт не должен выглядеть как официальный сайт ЖК или застройщика.
2. Нельзя использовать формулировки «официальный сайт», «от застройщика», «официальный представитель», «цены от застройщика» без проверки.
3. Нельзя обещать бронь, наличие квартир, цену и условия покупки без подтверждения.
4. Нельзя публиковать неподтверждённые данные по Аэродромной 18Г, Сенной 76 и другим объектам как факт.
5. Для старого корня нужно убрать впечатление сайта одного застройщика при переходе на новый портал.

## 6. SEO-аудит

### Сильные стороны

1. Домен точно отражает главный запрос: `novostroyki-borisoglebsk.ru`.
2. Уже подготовлена логика городского каталога.
3. Есть база для страниц объектов и будущих сравнений.
4. Есть потенциал собрать трафик по запросам:
   - новостройки Борисоглебска;
   - квартиры в новостройках Борисоглебск;
   - новые дома Борисоглебск;
   - купить квартиру в новостройке Борисоглебск;
   - Просторная 4А;
   - Аэродромная 18Г;
   - Сенная 76;
   - ПСК Борисоглебск.

### Слабые стороны

1. Корень сайта пока не оптимизирован под общий каталог.
2. Sitemap пока не соответствует новому домену.
3. Портальные страницы закрыты от индексации и не дают SEO-эффект.
4. Не хватает страниц застройщиков.
5. Не хватает базы объектов.
6. Не хватает сравнительных страниц и справочника покупателя.

## 7. Лидогенерация

### Уже хорошо

1. Новые формы фиксируют тип заявки и объект.
2. Формы аккуратно не обещают бронь и цену.
3. На портальных страницах есть сценарии подбора квартиры, а не только заявка на один ЖК.

2. Сделать разные формы под сценарии:
   - подбор новостройки;
   - сравнение объектов;
   - ипотека;
   - запрос планировок;
   - консультация по конкретному объекту.
3. Проверить, что все формы передают UTM, страницу заявки и источник.
4. Подготовить чек-лист обработки заявки менеджером.

## 9. Приоритетный вывод

Сайт уже можно развивать в сторону портала без переписывания всего проекта с нуля.

Правильная стратегия:

```text
1. Не ломать текущий работающий сайт.
2. Довести черновой каталог до готового состояния.
3. Расширить базу объектов.
4. Подготовить новую главную портала.
5. Только после проверки заменить корневой index.html, sitemap, robots и canonical.
```
