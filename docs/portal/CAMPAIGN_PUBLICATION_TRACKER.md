# Журнал рекламных размещений

Дата подготовки: 2026-07-14

## Назначение

Release pack содержит 11 готовых рекламных ссылок, но подготовленная ссылка не означает, что она уже опубликована.

Фактические размещения фиксируются отдельно:

```text
data/marketing/campaign-publications.json
```

В журнал добавляются только реально выполненные публикации.

Отсутствующая запись означает:

```text
not_published
```

## Начальное состояние

```text
Кампаний в release pack: 11
Фактических размещений: 0
Кампаний с размещениями: 0
Не опубликовано: 11
```

Это честное начальное состояние. Публикации во ВКонтакте, Telegram и на печатных носителях ещё не выполнялись.

## Структура записи

```json
{
  "publication_id": "portal_vk_catalog_2026_07_20_post_01",
  "campaign_id": "portal_vk_catalog",
  "status": "published",
  "channel": "vk",
  "external_url": "https://vk.com/wall-000000_000",
  "published_at": "2026-07-20T09:30:00+03:00",
  "owner": "Ответственный сотрудник",
  "evidence": "https://vk.com/wall-000000_000",
  "notes": "Пост в городском сообществе"
}
```

Пример приведён только для описания формата и не является фактической публикацией.

## Допустимые статусы

```text
published
paused
archived
```

Статус `planned` не записывается. Пока размещение не выполнено, запись отсутствует и отчёт показывает `not_published`.

## Обязательные поля

Каждая запись должна содержать:

```text
publication_id
campaign_id
status
channel
external_url
published_at
owner
evidence
```

## Правила проверки

### Campaign ID

`campaign_id` должен:

- существовать в release pack;
- относиться к активной UTM-кампании;
- использовать тот же канал, что и `utm_source`.

Например:

```text
portal_vk_catalog → channel=vk
portal_telegram_catalog → channel=telegram
portal_offline_qr → channel=offline
```

### Внешняя ссылка

`external_url` должна:

- использовать HTTPS;
- вести на реальную публикацию или страницу носителя;
- не содержать `lead_test`, `analytics_test` или `test_ack`;
- не содержать персональные параметры.

### Время

`published_at` хранится в ISO-формате с часовым поясом:

```text
2026-07-20T09:30:00+03:00
```

### Подтверждение

`evidence` может быть:

```text
https://...                    ссылка на публикацию или подтверждение
docs/...                       файл внутри репозитория
issue:123                      GitHub issue с материалами проверки
```

## Запрещённые поля

В журнале нельзя хранить:

```text
name
phone
email
client_fixation_id
realtor
realtor_id
manager
lead_payload
user_agent
```

Журнал отслеживает размещения, а не клиентов или сотрудников.

## Несколько публикаций одной кампании

Одна кампания может использоваться несколько раз.

Например, `portal_vk_catalog` может быть размещена:

- в основном городском сообществе;
- в тематической группе;
- в повторном посте через несколько недель.

Для каждого размещения создаётся отдельный уникальный `publication_id`.

Отчёт показывает:

- общее количество размещений;
- число кампаний хотя бы с одним размещением;
- последнюю публикацию по каждой кампании;
- текущий статус последней записи.

## Команды

Проверка журнала:

```bash
npm run validate:campaign-publications
```

Полный отчёт:

```bash
npm run marketing:publication-report
```

JSON:

```bash
npm run marketing:publication-report -- --format=json
```

CSV:

```bash
npm run marketing:publication-report -- --format=csv
```

Только Telegram:

```bash
npm run marketing:publication-report -- --channel=telegram
```

Только неопубликованные кампании:

```bash
npm run marketing:publication-report -- --status=not_published
```

## Порядок фактического размещения

1. Выбрать кампанию из `CAMPAIGN_RELEASE_PACK.md`.
2. Проверить актуальность рекламного текста и данных объекта.
3. Разместить готовую ссылку в выбранном канале.
4. Открыть опубликованный материал и проверить переход.
5. Добавить запись в `campaign-publications.json`.
6. Запустить validator и отчёт.
7. После накопления данных сверить клики и заявки по UTM.

## Что журнал не делает

- не публикует посты автоматически;
- не создаёт QR-изображения;
- не подключается к рекламным кабинетам;
- не меняет рабочую аналитику;
- не подтверждает доставку заявок;
- не заменяет CRM или календарь контента.
