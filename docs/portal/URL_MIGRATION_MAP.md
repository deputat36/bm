# Карта миграции URL портала novostroyki-borisoglebsk.ru

Дата обновления: 2026-07-13

Репозиторий: `deputat36/bm`.

## 1. Текущее состояние

Корень сайта уже переведён на независимый городской портал:

```text
/ = Новостройки Борисоглебска
```

Рабочая структура:

```text
/
/catalog/
/catalog/prostornaya-4a/
/catalog/aerodromnaya-18g/
/catalog/sennaya-76/
/developers/
/ipoteka/
/guides/
/news/
/contacts/
/privacy/
/spasibo/
```

Новые внутренние разделы сохраняют `noindex,follow`, пока данные, формы или материалы не прошли финальную проверку.

## 2. Главное правило миграции

```text
переходная страница → noindex,follow → проверенный целевой URL → контролируемый серверный редирект
```

Старый URL нельзя удалять или автоматически направлять на неподготовленную страницу.

## 3. Машиночитаемый реестр

Маршруты первой и второй очереди хранятся в:

```text
data/migration/legacy-routes.json
```

Проверка:

```bash
npm run validate:legacy
```

Для каждого маршрута фиксируются:

```text
source_url
source_file
target_url
target_file
target_href
status
redirect_phase
redirect_ready
blocking_reason
```

## 6. Почему серверные редиректы пока не включены

Редирект выпускается только после выполнения всех условий:

1. Целевой URL существует и открывается без ошибок.
2. На целевой странице нет неподтверждённых фактов.
3. Формы и системные ID заявок проверены.
4. Юридические формулировки прошли валидацию.
5. Принято решение об индексации целевого URL.
6. Определён способ настройки редиректов на хостинге.
7. Подготовлен план отката.
8. Проверено сохранение UTM-параметров.

До этого используются переходные HTML-страницы без `meta refresh` и JavaScript-перенаправления.

## 9. Правила sitemap и robots

В `sitemap.xml` нельзя включать:

```text
переходные legacy-URL
страницы со статусом requires_check
страницы с is_public_ready=false
служебные страницы заявок и благодарности
```

`robots.txt` должен указывать только актуальный sitemap нового домена.

## 10. Требования к будущим редиректам

Предпочтительный вариант — постоянный серверный редирект `301` или эквивалентное правило платформы.

Не использовать как финальное решение:

```text
meta refresh
JavaScript redirect
цепочки из нескольких редиректов
редирект всех старых URL только на главную
```

Каждый старый URL должен вести на наиболее близкую по смыслу страницу.

## 11. Контроль перед релизом

Для каждого маршрута проверить:

1. HTTP-статус старого URL.
2. Один переход без цепочки.
3. Финальный URL и canonical.
4. Отсутствие старого URL в sitemap.
5. Работу мобильной версии.
6. Сохранение UTM-меток.
7. Отсутствие циклического перенаправления.
8. Возможность быстро отменить правило.
