# Минимизация данных заявки

Дата обновления: 17 июля 2026 года.

## Цель

Не передавать вместе с заявкой технические данные, которые не нужны специалисту для связи, квалификации обращения и определения источника рекламы.

## Что изменено

- `user_agent` больше не формируется в `assets/js/main.js`;
- URL страницы передаётся только как origin и pathname;
- query-параметры и hash удаляются из URL страницы и referrer;
- ранее сохранённые first-touch и last-touch URL очищаются при следующем чтении атрибуции;
- tracking-параметры принимаются только из действующего allowlist;
- значения tracking-параметров очищаются от управляющих символов и ограничиваются 240 символами;
- форма, объект, источник, placement и разрешённые рекламные метки сохраняются.

## Что остаётся в заявке

- имя и телефон, введённые пользователем;
- выбранный объект и параметры покупки;
- идентификаторы формы и проекта;
- путь страницы без query-параметров;
- разрешённые UTM и рекламные идентификаторы;
- источник и placement;
- дата, согласие и технический ID фиксации обращения.

## Что не передаётся

- `navigator.userAgent`;
- произвольные query-параметры;
- hash страницы;
- произвольные ключи из URL;
- полный referrer с query-параметрами.

## Автоматическая защита

Проверка:

```bash
node tools/validate-lead-payload-privacy.mjs
```

Workflow:

```text
.github/workflows/lead-payload-privacy.yml
```

Guard проверяет:

- отсутствие сбора `user_agent`;
- очистку URL и referrer;
- обработку старых записей атрибуции;
- неизменность allowlist tracking-параметров;
- ограничение длины tracking-значений.

## Связь с запуском

Изменение закрывает автоматизируемую часть требования issue #71 об отсутствии `user_agent`, произвольных query-параметров и лишних локальных данных в фактической заявке. Оно не подтверждает реальную доставку письма и не заменяет ручную проверку на desktop, Android и iPhone.
