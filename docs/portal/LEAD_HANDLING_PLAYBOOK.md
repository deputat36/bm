# Операционная карта обработки заявок

Дата обновления: 2026-07-14

## Статус

```text
recommended_draft_not_operational
```

Документ описывает рекомендуемую логику первого контакта. Он не является утверждённым SLA, не назначает сотрудников, не подключает CRM и не меняет доставку заявок.

Источник правил:

```text
data/operations/lead-handling.json
```

## Зачем нужна карта

Техническая квалификация уже присваивает заявке статус:

```text
hot
warm
cold
```

Но одного статуса недостаточно. Менеджеру или специалисту нужно понимать:

```text
что сделать первым
что уточнить
чего не обещать
каким результатом завершить контакт
какое действие поставить следующим
```

## Приоритет обработки

### hot

Рекомендация:

```text
first_available_work_block
```

Это не обещание клиенту и не утверждённый норматив. Смысл — поставить обращение первым в доступной рабочей очереди.

Первый контакт должен завершиться одним из результатов:

```text
квалификация состоялась
согласовано следующее действие
назначен повторный контакт
нужна проверка источника
```

### warm

Рекомендация:

```text
same_working_day
```

Нужно определить, требуется ли:

```text
подбор объекта
консультация по конкретному адресу
предварительная ипотечная консультация
проверка сведений
повторный контакт
```

### cold

Рекомендация:

```text
planned_nurture_cycle
```

Холодная заявка не считается нецелевой только потому, что клиент заполнил короткую форму или пока не знает бюджет. Нужно определить горизонт покупки и безопасный следующий шаг.

Добавление в рекламные рассылки возможно только при соответствующем согласии.

## Роли форм

### primary

Короткая форма передаёт минимум данных. В первом разговоре уточняются только недостающие параметры:

```text
главный вопрос
объект или общий подбор
комнатность
способ покупки
срок
```

Имя, телефон и уже выбранный объект повторно запрашивать не нужно.

### detailed

Подробная форма уже содержит расширенный контекст. Задача специалиста:

```text
подтвердить полученные параметры
уточнить только пробелы
не проводить повторное анкетирование
перейти к предметному следующему шагу
```

## Типы заявок

### project_consultation

Первый разговор должен определить:

1. Что клиент хочет узнать по объекту.
2. Какой вариант квартиры рассматривает.
3. Как планирует покупку.
4. В какие сроки принимает решение.

Проверка первичного источника требуется по умолчанию.

Нельзя без актуального подтверждения сообщать как факт:

```text
цену
наличие
старт продаж
застройщика
сроки
планировки и характеристики
```

### portal_selection

Задача — сформировать профиль подбора:

```text
формат квартиры
район или адрес
бюджет
способ покупки
срок
критичные критерии
```

Каталог портала нельзя представлять как гарантированно полный рынок.

### mortgage

Собираются исходные данные для предварительной консультации:

```text
объект или бюджет
первоначальный взнос
предпочтительная программа
необходимость продажи своей недвижимости
```

Предварительная консультация не является:

```text
одобрением банка
гарантией ставки
гарантией программы
подтверждением соответствия объекта требованиям банка
```

## Объектные контексты

### Общий подбор

```text
object_id=all-newbuilds
source_check_required=false
```

Нужно определить критерии и выбрать подходящий маршрут. Нельзя обещать полноту рынка, гарантированное наличие или цену.

### Просторная 4А

```text
verification_status=partially_confirmed
source_check_required=true
```

Перед предметным ответом нужно повторно открыть первичный источник. Не подтверждены как текущие:

```text
цена
наличие
квартирография
последняя редакция документов
права на визуализации
```

### Аэродромная 18Г

```text
verification_status=requires_check
source_check_required=true
```

Первый контакт фиксирует интерес. Пока нет принятых источников, нельзя сообщать как подтверждённые:

```text
застройщика
цены
планировки
сроки
статус продаж
```

Можно предложить получение обновлений или сравнение с другими объектами.

### Сенная 76

```text
verification_status=requires_check
source_check_required=true
```

Действуют те же ограничения. При срочной покупке нужно предложить общий подбор, а не заполнять недостаток данных предположениями.

## Результаты первого контакта

Допустимые машинные статусы:

```text
contacted_qualified
contacted_follow_up_scheduled
no_answer_follow_up_planned
source_check_required
selection_started
mortgage_consultation_started
duplicate
invalid_or_spam
do_not_contact
```

Результаты с продолжением должны иметь следующее действие и дату возврата. Точные поля будущей CRM определяются отдельно.

## Следующие действия

```text
source_check
project_consultation
mortgage_precheck
selection_started
alternative_selection
updates_requested
follow_up_scheduled
close_no_action
```

## Генерация отчёта

```bash
node tools/build-lead-handling-playbook.mjs
node tools/build-lead-handling-playbook.mjs --format=json
node tools/build-lead-handling-playbook.mjs --format=csv
node tools/build-lead-handling-playbook.mjs --priority=hot
node tools/build-lead-handling-playbook.mjs --form-role=primary
node tools/build-lead-handling-playbook.mjs --lead-type=mortgage
node tools/build-lead-handling-playbook.mjs --object=prostornaya-4a
```

## Проверка

```bash
node tools/validate-lead-handling-playbook.mjs
```

Guard сверяет:

- пороги `hot/warm/cold` с кодом `main.js`;
- все 14 активных сценариев;
- роли `primary/detailed`;
- три активных типа заявки;
- четыре объектных контекста;
- статусы объектов с `data/projects/index.json`;
- отсутствие действующего SLA и назначения ответственных;
- отсутствие контактных данных в операционной спецификации.

## Что не реализовано

```text
реальное назначение ответственного
утверждённый SLA
автоматические задачи
история попыток связи
CRM-статусы
дедупликация по базе
контроль качества разговора
```

Эти функции требуют отдельного решения по backend и CRM.
