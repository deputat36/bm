# Проверка форм без реальной отправки

Дата обновления: 2026-07-15

## Включение режима

Dry-run включается только при двух параметрах URL:

```text
lead_test=dry-run
test_ack=1
```

Пример:

```text
https://novostroyki-borisoglebsk.ru/contacts/?lead_test=dry-run&test_ack=1#lead
```

На странице должна появиться заметная полоса «ТЕСТОВЫЙ РЕЖИМ — данные не отправляются». При отсутствии полосы тест не проводить.

Использовать только вымышленные тестовые данные, даже несмотря на обезличивание локального журнала.

## Что проверяется

- обязательные поля и согласие;
- системные `form_id`, `lead_type`, объект и ID объекта;
- наличие заполнения основных групп полей без сохранения их значений;
- мобильная панель и переход к форме;
- состояния доступности;
- переход на `/spasibo/`;
- тестовый ID вида `NB-TEST-...`;
- отсутствие реальной отправки и боевых событий аналитики.

## Что не проверяется

- доставка почты;
- работоспособность внешнего сервиса форм;
- будущая запись в CRM;
- серверные журналы;
- содержимое письма;
- сохранение введённых текстовых значений.

Для проверки доставки позже потребуется одна отдельно согласованная реальная тестовая заявка.

## Локальный журнал

Последние 20 обезличенных доказательств текущей вкладки сохраняются в:

```text
sessionStorage.newbuildsBorisoglebskLeadDryRuns
```

Запись содержит:

```text
client_fixation_id
lead_type
form_id
project_id
project_name
residential_complex
residential_complex_id
lead_source
placement
created_at
page_path
dry_run=true
delivery_status=not_sent
personal_data_stored=false
field_presence
```

`field_presence` содержит только логические признаки:

```text
name=true/false
phone=true/false
interest=true/false
purchase_method=true/false
timeline=true/false
comment=true/false
consent=true/false
```

В журнале и событии `newbuildLeadDryRun` не должны присутствовать введённые значения:

```text
имя
телефон
email
бюджет
комментарий
вопрос
срок
способ покупки
полный URL
tracking
полный payload формы
```

## Страница благодарности

В `localStorage.newbuildsBorisoglebskLastLead` сохраняется только технический контекст для `/spasibo/`:

```text
тестовый ID
тип заявки
форма
объект
статус test
время
```

Имя, телефон и текстовые ответы туда не попадают.

## Аналитический debug

При дополнительном включении:

```text
lead_test=dry-run
analytics_test=debug
test_ack=1
```

обезличенное событие используется для локального моделирования воронки. Внешние счётчики остаются отключёнными.

## Страницы для проверки

```text
/
/catalog/
/contacts/
/ipoteka/
/catalog/prostornaya-4a/
/catalog/aerodromnaya-18g/
/catalog/sennaya-76/
```

Для каждой страницы проверить пустую отправку, заполненную форму, выбор объекта, переход на страницу результата, ширины 320–430 px и клавиатурную навигацию.

После теста открыть DevTools и убедиться, что запись содержит только технические поля и `field_presence`.

## Завершение

Кнопка «Выключить тест» удаляет параметры режима и перезагружает страницу.

`sessionStorage` очищается браузером при закрытии вкладки. При необходимости журнал можно удалить вручную в DevTools.

## Автоматическая проверка

```bash
npm run validate:dry-run
```

Валидатор контролирует:

- необходимость двух параметров;
- остановку обычного обработчика формы;
- отсутствие сетевых запросов;
- тестовый ID;
- максимум 20 доказательств;
- `personal_data_stored=false`;
- только булевы признаки заполнения;
- запрет `records.push(payload)`;
- запрет полного payload во внутреннем событии;
- отключение рабочей аналитики;
- порядок подключения скриптов.
