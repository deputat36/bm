# Rate limit для заявок

Документ описывает ограничение частоты отправки заявок в Supabase Edge Function `newbuild-lead`.

## 1. Зачем нужен rate limit

Без ограничения частоты бот или скрипт может отправлять десятки заявок подряд. Это создаёт три проблемы:

- засоряет CRM и почту;
- мешает менеджеру быстро видеть настоящие заявки;
- искажает рекламную аналитику.

Поэтому в Edge Function добавлена серверная проверка частоты отправки.

## 2. Где реализовано

Файл:

```text
supabase/functions/newbuild-lead/index.ts
```

Таблица:

```text
public.newbuild_lead_rate_limits
```

SQL находится в:

```text
docs/supabase-leads.sql
```

## 3. Как считается лимит

Функция считает попытки за текущий час.

Технический fingerprint строится из:

- IP-адреса из серверных заголовков;
- user-agent;
- последних 4 цифр нормализованного телефона.

IP не хранится в открытом виде. В таблицу записывается хеш fingerprint.

## 4. Значение лимита

По умолчанию используется 12 попыток в час.

Для реального сайта это достаточно мягкое ограничение:

- обычный покупатель не отправит 12 заявок за час;
- менеджер при тестах сможет проверить несколько форм;
- бот быстро упрётся в лимит.

## 5. Что происходит при превышении

Если лимит превышен, функция возвращает мягкую блокировку:

```json
{
  "success": true,
  "blocked": true,
  "reason": "rate_limit"
}
```

Такая заявка не попадает в основную таблицу заявок, но попытка фиксируется в таблице rate limit.

Важно: ответ специально выглядит как успешный. Это сделано, чтобы бот не понимал, что именно его заблокировало.

## 6. Какие данные хранит таблица

`newbuild_lead_rate_limits` хранит:

- `fingerprint` — технический хеш;
- `window_start` — начало часового окна;
- `attempt_count` — количество попыток;
- `first_attempt_at` — первая попытка;
- `last_attempt_at` — последняя попытка;
- `last_payload` — последняя техническая заявка для диагностики.

## 7. Как смотреть подозрительные попытки

Пример запроса:

```sql
select
  window_start,
  attempt_count,
  last_attempt_at,
  last_payload->>'lead_type' as lead_type,
  last_payload->>'phone' as phone
from public.newbuild_lead_rate_limits
order by last_attempt_at desc
limit 50;
```

## 8. Когда менять лимит

Снижать лимит стоит, если:

- пошёл спам;
- в таблице много повторов за короткое время;
- заявки идут с одинаковыми телефонами или похожими данными.

Повышать лимит стоит, если:

- менеджеры активно тестируют формы;
- есть киоск/планшет в офисе, где несколько клиентов подряд оставляют заявки с одного устройства;
- планируется офлайн-мероприятие с QR-кодом.

## 9. Что добавить позже

- отдельный лимит по телефону;
- отдельный лимит по IP;
- белый список для офисного планшета;
- чёрный список подозрительных fingerprint;
- ежедневную очистку старых записей rate limit;
- уведомление администратору при массовых блокировках.
