# Правила качества JSON-данных

Дата фиксации: 2026-07-05

## 1. Назначение

JSON-файлы в папке `data` постепенно становятся основой будущего каталога, фильтров, сравнения, sitemap и интеграции заявок с CRM.

Поэтому данные должны быть проверяемыми, аккуратными и юридически безопасными.

## 2. Техническая проверка

Добавлен простой валидатор синтаксиса JSON:

```text
tools/validate-json.mjs
```

Запуск:

```bash
npm run validate:json
```

В GitHub Actions проверка запускается отдельным шагом после HTML-валидации.

## 3. Общие правила

1. Все JSON-файлы должны быть валидными.
2. Все `id` должны быть стабильными и не меняться без причины.
3. Для новых объектов сначала создаётся подробный файл, затем запись в индексном файле.
4. Нельзя удалять `id`, если на него уже могут ссылаться страницы, заявки или будущая CRM.
5. Не подтверждённые данные нужно помечать отдельным статусом или `is_verified: false`.

## 4. Проекты

Для каждого ЖК должны быть:

```text
/data/projects/{slug}.json
/data/projects/index.json
```

Обязательные поля в индексе:

```text
id
slug
name
city
address
builder_id
status
sales_status
detail_url
data_file
is_active
```

`builder_id` должен совпадать с ID застройщика из `/data/builders/index.json`.

## 5. Застройщики

Для каждого застройщика должны быть:

```text
/data/builders/{slug}.json
/data/builders/index.json
```

Обязательные поля в индексе:

```text
id
slug
name
detail_url
data_file
is_active
```

## 6. Квартирография

Для каждого ЖК можно создавать файл:

```text
/data/apartment-types/{project}.json
```

Обязательные поля:

```text
project_id
type
title
rooms
count
landing_url
is_active
```

Площади должны быть указаны через `area_min` и `area_max`. Если площадь одна, оба значения могут совпадать.

## 7. Планировки

Для каждого ЖК можно создавать файл:

```text
/data/layouts/{project}.json
```

Обязательные поля:

```text
id
project_id
apartment_type
title
rooms
count
description
is_verified
is_active
```

Для площади нужно указать одно из:

```text
area
area_min + area_max
```

Если `is_verified = false`, планировку нельзя публиковать как официальную схему. Её можно использовать только как черновую внутреннюю структуру.

## 8. Документы

Для каждого ЖК можно создавать файл:

```text
/data/documents/{project}.json
```

Обязательные поля:

```text
id
project_id
title
type
number
date
source_name
is_verified
is_active
```

Если `is_verified = true`, у документа должна быть ссылка на источник или локальный файл:

```text
source_url
local_file
```

## 9. Страницы

Индекс страниц:

```text
/data/pages/index.json
```

Если страница имеет статус `draft`, у неё должно быть:

```text
robots: "noindex,follow"
```

После ручной проверки статус можно менять:

```text
ready
published
```

В sitemap можно добавлять только страницы со статусом `published`.

## 10. Что нельзя делать

1. Нельзя публиковать цены как факт без подтверждения.
2. Нельзя публиковать неподтверждённые планировки как официальные.
3. Нельзя создавать впечатление, что портал является официальным сайтом застройщика.
4. Нельзя удалять старые URL и данные без карты редиректов и проверки заявок.
5. Нельзя включать черновики в sitemap.
